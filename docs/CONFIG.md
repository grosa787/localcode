# Configuration

LocalCode persists user-level configuration as TOML at
`~/.localcode/config.toml`. The schema is owned by
[`src/config/types.ts`](../src/config/types.ts) (Zod) and read/written
through [`src/config/config-manager.ts`](../src/config/config-manager.ts).

## Where the file lives

| Layer | Path | Purpose |
| --- | --- | --- |
| Per-user | `~/.localcode/config.toml` | Backend, model, permissions, context, sound. |
| Per-user | `~/.localcode/sessions.db` | SQLite session/message store. |
| Per-user | `~/.localcode/skills/*.md` | Global fallback skills. |
| Per-user | `~/.localcode/skills-active.json` | Global active-state fallback. |
| Per-project | `<projectRoot>/.localcode/LOCALCODE.md` | Generated by `/init`. |
| Per-project | `<projectRoot>/.localcode/skills/*.md` | Project-local skills (priority). |
| Per-project | `<projectRoot>/.localcode/skills-active.json` | Project active-state. |

There is **no** project-level `config.toml`. Backend, model, and
context settings are user-wide so a single LocalCode install can serve
many projects without juggling per-project setup.

## Atomic writes

`ConfigManager.write` validates the payload against `ConfigSchema`,
serialises with `smol-toml`, writes to `<file>.tmp`, and `rename()`s
on top of the real path. Failed renames clean up the temp file
(best-effort) so a half-finished write doesn't break subsequent reads.

`ConfigManager.update(partial)` does a deep merge under the hood
(`deepMerge` in `config-manager.ts` — recurses on plain objects,
replaces arrays/scalars). The merged result is re-validated before
writing.

## Schema

```toml
# All fields shown with their defaults from src/config/defaults.ts.

[backend]
type = "ollama"          # "ollama" | "lmstudio"
baseUrl = "http://localhost:11434"

[model]
current = ""             # populated by onboarding
available = []           # populated by /v1/models scan

[onboarding]
completed = false        # set true by OnboardingScreen

[permissions]
autoApprove = []         # subset of: "read_file", "write_file",
                         #            "run_command", "list_dir", "glob_search"

[context]
maxTokens = 8192         # forwarded to Ollama as num_ctx (1024 .. 1_048_576)
keepAliveSeconds = 1800  # Ollama keep_alive (0 .. 86_400)

[sound]
enabled = false          # master switch
onCompletion = true
onApproval = true
onError = true
volume = 0.5             # 0.0 .. 1.0
completionFile = ""      # absolute path to a .wav/.mp3, or "" → terminal bell
approvalFile = ""
errorFile = ""
```

## Field reference

### `[backend]`

| Field | Type | Default (Ollama) | Default (LM Studio) | Notes |
| --- | --- | --- | --- | --- |
| `type` | `"ollama" \| "lmstudio"` | `"ollama"` | `"lmstudio"` | Determines which native fallback URL the adapter tries (`/api/tags` for Ollama). |
| `baseUrl` | non-empty string | `http://localhost:11434` | `http://localhost:1234/v1` | Trailing `/v1` is stripped/joined as needed by `LLMAdapter.joinUrl`. |

The adapter's `resolveBackend()` falls back to URL inspection when
`type` isn't set explicitly, so a custom URL on `:11434` is treated as
Ollama; everything else as LM Studio.

### `[model]`

| Field | Type | Notes |
| --- | --- | --- |
| `current` | string | Active model name, e.g. `qwen2.5-coder:32b`. Empty until onboarding. |
| `available` | `string[]` | Cache of `/v1/models` results. Refreshed by `/model refresh`. |

`--model <name>` overrides `model.current` for the duration of the
process without persisting.

### `[onboarding]`

| Field | Type | Notes |
| --- | --- | --- |
| `completed` | boolean | When `true`, `cli.tsx` skips `OnboardingScreen` unless `--reconfigure` is passed. |

### `[permissions]`

| Field | Type | Notes |
| --- | --- | --- |
| `autoApprove` | `string[]` | Tool names the executor may run without prompting. |

The Zod enum (`AutoApprovableToolSchema`) only accepts the five tool
names that have a meaningful approval path: `read_file`, `write_file`,
`run_command`, `list_dir`, `glob_search`. Read-only tools are pre-allow
so leaving them out of the list has no effect; the schema accepts them
to match `AutoApprovableTool` from `types/global.d.ts`. Practically,
users only ever add `write_file` and `run_command`.

### `[context]`

| Field | Type | Range | Notes |
| --- | --- | --- | --- |
| `maxTokens` | int | 1024 .. 1_048_576 | Forwarded to Ollama as `options.num_ctx`. LM Studio ignores this; the value is then advisory and only governs LocalCode's own message-history budgeting. |
| `keepAliveSeconds` | int | 0 .. 86_400 | Ollama `keep_alive` (sent as `"<N>s"`). 0 unloads the model immediately. LM Studio ignores it. |

Both fields rebuild `LLMAdapter` when changed (`App.useMemo` keys on
both, and the rebuild aborts any in-flight stream).

### `[sound]`

| Field | Type | Notes |
| --- | --- | --- |
| `enabled` | bool | Master switch. When `false`, no cues fire. |
| `onCompletion` | bool | Trigger on `streamChat.onDone` without error. |
| `onApproval` | bool | Trigger when `ApprovalPrompt` appears. |
| `onError` | bool | Trigger on stream error or tool-result `success: false`. |
| `volume` | number | 0.0 (silent) .. 1.0 (full). Used as `afplay -v <volume>` on macOS. |
| `completionFile` | string \| null | Absolute path to a `.wav` / `.mp3`. `null` (or `""`) → terminal bell. |
| `approvalFile` | string \| null | Same, for approval cue. |
| `errorFile` | string \| null | Same, for error cue. |

Playback strategy (`src/integration/sound.ts`):

- macOS: `afplay -v <volume> <file>` when a file is set, else bell.
- Linux: `aplay <file>` when a file is set, else bell.
- Other platforms (incl. Windows/WSL): always the terminal bell
  (`\x07`).
- The `getConfig` thunk is read fresh on every `play()`, so live
  config edits take effect without restart.

## Editing config

Three supported paths, all validated:

1. **Onboarding** — first launch (or `--reconfigure`). Picks backend,
   URL, model, then writes the full default tree.
2. **Slash commands** — `/permissions`, `/ctxsize`, `/provider`, and
   `/model` route through `ConfigManager.update(...)`. They never
   bypass schema validation.
3. **Hand-edit** — `vi ~/.localcode/config.toml` works, but a typo will
   raise `ConfigValidationError` on next read; recover by running
   `localcode --reconfigure` or fixing the file.

The model can also propose a config edit through the `edit_file` tool
(see `## Self-configuration` in the system prompt) — the user still
sees a diff before any change lands.

## Defaults helper

`src/config/defaults.ts` exposes:

- `DEFAULTS` — frozen literal of every default value.
- `getDefaultBaseUrl(backend)` — used by onboarding.
- `getMaxContextTokens(backend)` — used by `App.contextPercent`'s
  fallback when `maxTokens` is 0 (which the schema doesn't allow, but
  the defensive code paths still check).
- `getDefaultConfig(backend)` — emits a brand-new schema-valid
  `Config` for the chosen backend.

## Errors

`ConfigManager` throws:

- `ConfigReadError` — file missing or unreadable, or TOML parse
  failure.
- `ConfigValidationError` — TOML parses but Zod rejects the result.
  Carries the original `issues` array.
- `ConfigWriteError` — disk full, permission denied, etc.

`App.useEffect` catches read errors and renders a red "Failed to load
config" splash with a hint to re-run onboarding.

## Migrating an old config

Each Zod sub-schema has `.default(...)` so missing sections are filled
in transparently:

- `[permissions]` defaults to `{ autoApprove: [] }`.
- `[context]` defaults to `{ maxTokens: 8192, keepAliveSeconds: 1800 }`.
- `[sound]` defaults to the everything-off / per-event-on shape.

A pre-Round-2 config that lacks `[permissions]` and `[context]` reads
fine; the next `update()` writes the gaps back. Pre-Round-3 configs
that lack `[sound]` are filled in the same way.
